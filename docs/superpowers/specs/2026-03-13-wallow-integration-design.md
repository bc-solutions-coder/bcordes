# Wallow Backend Integration Design

**Date:** 2026-03-13
**Status:** Draft
**Goal:** Replace the existing ORPC/Drizzle/Better Auth backend with Wallow, a C# ASP.NET Core API, using a BFF (Backend-for-Frontend) pattern through TanStack Start's server capabilities.

---

## Context

The personal site (bcordes.dev) currently has a self-contained backend:

- **ORPC** type-safe RPC layer with direct PostgreSQL access via Drizzle ORM
- **Better Auth** for email/password session-based authentication
- **Local PostgreSQL** with tables for contacts, page views, project reactions, subscribers, and auth

Wallow is a multi-tenant C# backend running at `http://localhost:5000` with:

- Keycloak OIDC authentication (realm: `wallow`)
- RESTful API at `/api/v1/{module}/{resource}`
- SignalR real-time at `/hubs/realtime`
- Modules: Identity, Showcases, Inquiries, Notifications (and more)

**Tenant context:** bcordes.dev is the main tenant and platform. No multi-tenant complexity is needed on the frontend — the site operates under a single tenant context.

### Wallow Inquiries Module — Current State

The Inquiries module is minimal. What exists today:

**Endpoints:**

- `POST /api/v1/inquiries` — anonymous submit
- `GET /api/v1/inquiries` — list all (optional `?status=` filter)
- `GET /api/v1/inquiries/{id}` — get by ID
- `PUT /api/v1/inquiries/{id}/status` — update status

**Entity fields:** Name, Email, Company, ProjectType, BudgetRange, Timeline, Message, Status, SubmitterIpAddress

**What does NOT exist yet:**

- No comments/notes on inquiries
- No `/mine` endpoint (query by authenticated user's email)
- No SignalR events for inquiries (`IRealtimeDispatcher` not injected)
- No `inquiries.read` or `inquiries.write` in `ApiScopes.ValidScopes`
- No email-to-account linking on registration
- `showcases.read` exists in `PermissionExpansionMiddleware` but is not in `ValidScopes`

These features are documented as **Wallow backend prerequisites** throughout this spec. The frontend integration can proceed with what exists, and enhanced features can be added as Wallow evolves.

## Architecture: BFF Pattern

```
Browser (React)
    |
    | (same-origin requests, HTTP-only cookies)
    | (WebSocket proxied through BFF for SignalR)
    v
TanStack Start / Nitro Server (BFF)
    |
    | (server-to-server, Bearer JWT)
    | (WebSocket proxy to SignalR hub)
    v
Wallow API (http://localhost:5000)
    |
    v
Keycloak (http://localhost:8080)
```

The TanStack Start server acts as a Backend-for-Frontend:

- Handles Keycloak OIDC auth flow server-side
- Stores tokens in encrypted HTTP-only cookies
- Proxies API calls to Wallow with the user's access token
- Proxies SignalR WebSocket connections (JWT never exposed to browser)
- Enables full SSR with server-side data fetching

The browser **never** sees or handles JWTs. All token management is server-side, including SignalR connections which are proxied through the BFF.

## 1. Authentication

### OIDC Authorization Code Flow (Server-Side)

1. **Login**: User clicks login -> server redirects to Keycloak authorization endpoint
2. **Authenticate**: User enters credentials on Keycloak login page
3. **Callback**: Keycloak redirects back to `/auth/callback` with authorization code
4. **Token Exchange**: Server exchanges code for tokens (access, refresh, ID token) using client secret
5. **Session**: Tokens stored in encrypted HTTP-only cookie
6. **Refresh**: Server transparently refreshes tokens before expiry

### Keycloak Client

Register a **new confidential client** in Keycloak specifically for the BFF:

| Parameter                       | Value                                                             |
| ------------------------------- | ----------------------------------------------------------------- |
| Client ID                       | `wallow-bff`                                                      |
| Client Type                     | Confidential (with client secret)                                 |
| Client Secret                   | Generated by Keycloak, stored in `KEYCLOAK_CLIENT_SECRET` env var |
| Valid Redirect URIs             | `http://localhost:3000/auth/callback` (dev), production URL       |
| Valid Post Logout Redirect URIs | `http://localhost:3000/` (dev), production URL                    |
| Web Origins                     | `http://localhost:3000`                                           |

This is separate from `wallow-spa` (the public SPA client). The BFF uses a confidential client because it can securely store the client secret server-side, providing stronger security than PKCE alone.

### OIDC Configuration

| Parameter              | Value                                       |
| ---------------------- | ------------------------------------------- |
| Issuer                 | `http://localhost:8080/realms/wallow`       |
| Client ID              | `wallow-bff`                                |
| Client Secret          | Required (confidential client)              |
| Authorization Endpoint | `{issuer}/protocol/openid-connect/auth`     |
| Token Endpoint         | `{issuer}/protocol/openid-connect/token`    |
| Userinfo Endpoint      | `{issuer}/protocol/openid-connect/userinfo` |
| End Session Endpoint   | `{issuer}/protocol/openid-connect/logout`   |
| Scopes                 | `openid profile email`                      |
| Response Type          | `code`                                      |
| Grant Type             | `authorization_code`                        |

### Session Cookie

- Encrypted using `SESSION_SECRET` (AES-256-GCM via Web Crypto API)
- HTTP-only, Secure (in production), SameSite=Strict
- Contains: access token, refresh token, token expiry, user info (name, email, roles, permissions, tenant), session version (for concurrent refresh protection)

#### Concurrent Refresh Protection

In SSR, multiple route loaders may execute in parallel and detect an expired token simultaneously. Since Keycloak refresh tokens are single-use, only one refresh attempt can succeed. To prevent race conditions:

- Use a per-session mutex (in-memory lock keyed by session ID) around token refresh
- First loader to acquire the lock performs the refresh and updates the session
- Subsequent loaders wait for the lock and use the already-refreshed token
- The session cookie includes a `version` field incremented on each refresh to detect stale reads

### Library Choice

Use `arctic` (by the lucia-auth author) for the OIDC flow. It's a lightweight, zero-dependency OAuth2/OIDC helper designed for server-side use. Alternatively, `openid-client` (the more established library) works too. Both support authorization code flow with client secret.

For session encryption, use `iron-webcrypto` or a simple AES-GCM implementation with the Web Crypto API (available in Nitro's runtime).

### Routes

| Route            | Method | Purpose                                                          |
| ---------------- | ------ | ---------------------------------------------------------------- |
| `/auth/login`    | GET    | Generate state + nonce, redirect to Keycloak                     |
| `/auth/callback` | GET    | Exchange code for tokens, set session cookie, redirect to app    |
| `/auth/logout`   | POST   | Clear session cookie, redirect to Keycloak logout                |
| `/auth/me`       | GET    | Return current user info from session (used by `useUser()` hook) |

**Note:** Logout is POST-only to prevent CSRF attacks (a GET logout could be triggered by `<img src="/auth/logout">`).

### CSRF Protection

- `SameSite=Strict` on session cookies prevents most cross-site request scenarios
- The `/auth/login` route uses a `state` parameter (OIDC standard) to prevent CSRF on the auth flow
- For mutations via server functions: TanStack Start server functions use POST by default and validate the `Origin` header against the expected origin
- The `/auth/logout` route requires POST method, preventing image/link-based CSRF

### User Info Exposed to Client

The `useUser()` hook returns (from route loader context or `/auth/me`):

```typescript
interface User {
  id: string
  name: string
  email: string
  roles: string[] // ["admin", "manager", "user"]
  permissions: string[] // ["InvoicesRead", "InvoicesWrite", ...] — expanded from roles
  tenantId: string // from organization claim
  tenantName: string
}
```

The `permissions` array is populated from the JWT claims after Wallow's `PermissionExpansionMiddleware` processes the token. This enables fine-grained client-side permission checks (e.g., showing/hiding UI elements based on specific permissions rather than broad roles).

### Protected Routes

Server-side middleware checks for a valid session cookie. If missing or expired (and refresh fails), redirect to `/auth/login` with a `returnTo` parameter.

For client-side route guards, `useUser()` returns `null` for unauthenticated users. Protected route components redirect to login.

## 2. Wallow API Client

### Server-Side HTTP Client

A server-only module (`src/lib/wallow/client.ts`) that:

1. Takes the user's access token from the session
2. Makes HTTP requests to Wallow's REST API
3. Handles errors (RFC 7807 Problem Details)
4. Retries on 401 (refresh token via session mutex, retry once)
5. Backs off on 429 (respect `Retry-After` header)

```typescript
// Conceptual API
const wallow = createWallowClient({
  baseUrl: process.env.WALLOW_API_URL, // http://localhost:5000
  getAccessToken: () => getSessionToken(request),
})

// Usage in server functions
const showcases = await wallow.get('/api/v1/showcases')
const inquiry = await wallow.post('/api/v1/inquiries', { body: data })
```

### Error Handling

Wallow returns RFC 7807 Problem Details:

```typescript
interface ProblemDetails {
  type: string
  title: string
  status: number
  detail: string
  traceId: string
  code: string
  errors?: Record<string, string[]> // validation errors
}
```

The client maps these to typed errors that server functions can return to the frontend for user-facing error messages (e.g., validation errors shown inline on forms).

### Service Account Client

For unauthenticated/public operations (e.g., submitting a contact inquiry without login), a separate service account client uses OAuth2 client credentials:

```typescript
const serviceClient = createWallowServiceClient({
  baseUrl: process.env.WALLOW_API_URL,
  clientId: process.env.WALLOW_SA_CLIENT_ID,
  clientSecret: process.env.WALLOW_SA_CLIENT_SECRET,
  tokenEndpoint: `${process.env.KEYCLOAK_URL}/realms/wallow/protocol/openid-connect/token`,
})
```

This client caches tokens and refreshes 30 seconds before expiry. Used server-side only.

#### Required Service Account Scopes

The service account needs the following scopes provisioned in Wallow for public operations:

| Scope             | Status                                                                          | Purpose                                          |
| ----------------- | ------------------------------------------------------------------------------- | ------------------------------------------------ |
| `showcases.read`  | In PermissionExpansionMiddleware but NOT in ValidScopes — **needs to be added** | Fetch showcases for SSR (public pages)           |
| `inquiries.write` | **Does not exist — needs to be created**                                        | Submit inquiries on behalf of anonymous visitors |

**Wallow backend prerequisite:** Both scopes must be added to `ApiScopes.ValidScopes` before the service account can use them. The `showcases.read` mapping already exists in the middleware but isn't registered as a valid scope.

## 3. Data Fetching

### Server Functions

Each Wallow API interaction is wrapped in a `createServerFn()`:

#### Showcases

```typescript
// Public — uses service account for SSR
const fetchShowcases = createServerFn({ method: 'GET' }).handler(async () => {
  return serviceClient.get('/api/v1/showcases')
})

const fetchShowcase = createServerFn({ method: 'GET' }).handler(
  async ({ data: slug }) => {
    return serviceClient.get(`/api/v1/showcases/${slug}`)
  },
)
```

**Note:** Showcase endpoint paths need to be confirmed against the actual Wallow controller. The paths above follow the standard convention.

#### Inquiries

Inquiries support two submission modes:

1. **Anonymous**: Visitor submits without login. The BFF uses the service account to call `POST /api/v1/inquiries`. The inquiry is stored with the visitor's email.
2. **Authenticated**: Logged-in user submits. The BFF uses the user's session token. The inquiry is stored with both the email and the authenticated user context.

```typescript
// Submit inquiry — BFF determines which client to use based on session
const submitInquiry = createServerFn({ method: 'POST' }).handler(
  async ({ data }) => {
    const session = await getSession(request)
    if (session) {
      // Authenticated: use user's token
      return wallow.post('/api/v1/inquiries', { body: data })
    }
    // Anonymous: use service account
    return serviceClient.post('/api/v1/inquiries', { body: data })
  },
)

// Protected — admin: fetch all inquiries (with optional status filter)
const fetchInquiries = createServerFn({ method: 'GET' }).handler(
  async ({ data }) => {
    const params = data?.status ? `?status=${data.status}` : ''
    return wallow.get(`/api/v1/inquiries${params}`)
  },
)

// Protected — admin: get single inquiry
const fetchInquiry = createServerFn({ method: 'GET' }).handler(
  async ({ data: id }) => {
    return wallow.get(`/api/v1/inquiries/${id}`)
  },
)

// Protected — admin: update inquiry status
const updateInquiryStatus = createServerFn({ method: 'POST' }).handler(
  async ({ data }) => {
    return wallow.put(`/api/v1/inquiries/${data.id}/status`, {
      body: { status: data.status },
    })
  },
)
```

#### User Inquiry View (Future Enhancement)

Currently, there is no endpoint for users to view their own inquiries. When Wallow adds a `/mine` or email-filtered endpoint, the BFF will add:

```typescript
// Future: Authenticated user views their own inquiries
const fetchMyInquiries = createServerFn({ method: 'GET' }).handler(async () => {
  return wallow.get('/api/v1/inquiries/mine') // endpoint does not exist yet
})
```

**Wallow backend prerequisites for user inquiry view:**

1. Add `GET /api/v1/inquiries/mine` endpoint (or `GET /api/v1/inquiries?email={email}` with auth scoping)
2. Optionally: link anonymous inquiries to accounts by email on registration
3. Add `inquiries.read` scope for users who should see their own inquiries

Until these exist, the `/dashboard/inquiries` page is **deferred**.

#### Notifications

```typescript
// Protected
const fetchNotifications = createServerFn({ method: 'GET' }).handler(
  async () => {
    return wallow.get('/api/v1/notifications')
  },
)
```

### Route Loaders (SSR)

```typescript
export const Route = createFileRoute('/work/')({

```

## 4. SignalR Real-Time

### Connection Strategy: WebSocket Proxy Through BFF

To maintain the security guarantee that JWTs never reach the browser, SignalR WebSocket connections are **proxied through the BFF**. The Nitro server upgrades the client's WebSocket connection and establishes a corresponding connection to Wallow's hub, forwarding messages bidirectionally.

```
Browser (WebSocket)           Nitro/BFF (WebSocket Proxy)           Wallow
    |                              |                                   |
    |--- ws://localhost:3000 ----->|                                   |
    |    /api/ws/realtime          |--- ws://localhost:5000 ---------->|
    |    (cookie auth)             |    /hubs/realtime                 |
    |                              |    (Bearer JWT from session)      |
    |<--- messages forwarded ----->|<--- messages forwarded ---------->|
```

#### Implementation

Create a WebSocket proxy endpoint at `/api/ws/realtime`:

1. Client connects to `ws://localhost:3000/api/ws/realtime` (authenticated via session cookie)
2. BFF validates the session, extracts the access token
3. BFF opens a WebSocket connection to `ws://localhost:5000/hubs/realtime` with `Authorization: Bearer <token>` via query string (SignalR convention)
4. BFF forwards messages bidirectionally between client and Wallow
5. On token refresh, BFF can reconnect to Wallow transparently

Nitro supports WebSocket handling via `defineWebSocketHandler()` (built on crossws/H3). The proxy logic:

- On `open`: validate session, connect to Wallow hub, join tenant group
- On `message`: forward from client to Wallow and vice versa
- On `close`: disconnect Wallow connection, clean up
- On Wallow disconnect: attempt reconnect with refreshed token

#### Tenant Group Join

On connection open, the BFF must automatically join the user's tenant group:

```typescript
// After connecting to Wallow's hub
await wallowConnection.invoke('JoinGroup', `tenant:${user.tenantId}`)
```

This is required because Wallow dispatches tenant-scoped notifications to `tenant:{tenantId}` groups. Without joining, `BroadcastToTenantAsync` messages will never reach the client. The group must also be re-joined after any reconnection.

### TypeScript Types

```typescript
/** Envelope for all SignalR messages from Wallow */
interface RealtimeEnvelope {
  type: string // e.g., "NotificationCreated"
  module: string // e.g., "Notifications"
  payload: unknown // event-specific data
  timestamp: string // UTC ISO 8601
  correlationId?: string
}
```

### Events to Listen For

Only Notifications currently emit SignalR events. Inquiries do **not** have SignalR integration.

| Method                 | Events                                            | Purpose                                |
| ---------------------- | ------------------------------------------------- | -------------------------------------- |
| `ReceiveNotifications` | `NotificationCreated`, `AnnouncementPublished`    | New notification/announcement for user |
| `ReceivePresence`      | `UserOnline`, `UserOffline`, `PageViewersUpdated` | Optional presence tracking             |

**Future:** If `IRealtimeDispatcher` is added to the Inquiries module, inquiry status change events could be surfaced to users in real-time.

### React Hook

```typescript
function useSignalR() {
  // Connects to ws://localhost:3000/api/ws/realtime (proxied)
  // Automatically reconnects on disconnect
  // Re-joins tenant group after reconnection
  // Exposes typed subscribe/unsubscribe for RealtimeEnvelope events
  // Cleans up WebSocket on component unmount
  // Returns connection status for UI indicators
}
```
